## 2. 라우트별 코드 스플리팅 현황
`src/App.tsx`에서 주요 페이지는 React.lazy 로 스플릿되어 있음:
- Lazy 로드: `HomePageWithRedirect`, `NoticePage`, `CommunityPage`, `ContactPage`, `MyPage`, `AdminPage`, `TemplateEditorPage`, `AllPagesListPage`, `FavoritesPage`, `TemplatesGalleryPage`.
- 즉시 로드: `react-vendor`, `firebase` 청크 및 공용 컨텍스트/유틸.

### Lazy-load 후보 (추후 검토)
- **Firebase 의존 모듈**: 현재 초기 번들에 포함되어 `firebase-BjjBD64E.js` (~494 kB). 인증/데이터가 필요한 라우트 진입 시점으로 지연 로딩하는 방안 검토 필요.
- **Admin/MyPage 관련 대형 컴포넌트**: 이미 청크 분리되어 있으나, 내부 위젯/툴 모음(Firebase SDK 포함)이 초기화 시 전부 로드되는지 추가 추적 필요.
- **공용 CSS**: `index.css`가 156 kB에 달해, 페이지별 스타일 분리 가능성 검토.

## 3. 이미지/폰트 로딩 타임라인
- 현재 dist 산출물 기준 거대한 이미지/폰트 파일이 없어 정밀 타임라인 계측 결과가 없음.
- 추후 실 서비스 환경(예: Sentry/Chrome DevTools Performance)에서 네트워크 타임라인 캡처를 권장.

## 4. Lighthouse 측정 (실행 보류)
- 로컬 환경에서 Headless Chrome 실행 권한이 확인되지 않아 Lighthouse 자동화를 수행하지 못함.
- 다음 단계에서 `npm run dev -- --host`로 서버를 띄운 뒤 아래 페이지 측정을 권장:
  - 홈(`/`)
  - 템플릿 갤러리(`/templates`)
  - MyPage(`/mypage`)
- 목표: **Performance 90+** (현재 값 미측정, “추후 목표”로 유지).

## 5. 요약 및 후속 액션
1. Firebase SDK(약 0.5 MB)가 초기 번들에 포함 → 라우트 단위 지연 로딩 방안 조사.
2. CSS/공용 번들(`index-*.js`, `index-*.css`) 가중치를 줄이기 위해 사용량 분석 필요.
3. Lighthouse 측정 환경 구성(Chrome 설치/CI 빌드) 후 정기 점검 체계 마련.
4. 이미지/폰트 자산이 늘어날 경우, DevTools 로 네트워크 타임라인 캡처 병행.
